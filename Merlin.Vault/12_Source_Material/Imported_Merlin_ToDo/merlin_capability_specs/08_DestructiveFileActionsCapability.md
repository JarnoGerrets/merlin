# 08 - Destructive File Actions Capability

## Goal

Design a safe destructive-file-action capability for delete, recycle, restore, cleanup, and possibly permanent deletion later. This must be the last missing capability implemented because mistakes here can destroy user data.

## Current state

`destructive_file_action` exists as an unsupported capability domain. The current missing message correctly says Merlin does not support destructive file actions to protect data from accidental or unsafe changes.

## User value

Example requests:

- "Move this duplicate file to the recycle bin."
- "Clean up old logs from this test folder."
- "Delete the empty folders in this project."
- "Remove the temporary files Merlin created."
- "Restore the file you just deleted."

## Core principle

Start with reversible actions only.

The first implementation should not permanently delete files. It should move items to the Recycle Bin or a Merlin quarantine folder with restore metadata.

## Scope

### Phase 1: Detection and refusal

- Detect delete/wipe/format requests.
- Explain unsupported state.
- Route safely instead of falling into general conversation.

### Phase 2: Recycle-bin staging

- Allow moving specific files to Recycle Bin after hard confirmation.
- Only within previously approved folders.
- No wildcards.
- No recursive folder deletion unless explicitly enabled later.

### Phase 3: Merlin quarantine

- Move files to a quarantine folder.
- Store original path.
- Support restore.
- Good for project cleanup.

### Phase 4: Cleanup recipes

- Safe cleanup of Merlin-generated temp/cache files.
- Dry-run first.
- Confirm exact file count and size.

### Phase 5: Permanent delete

- Probably never needed for first product stage.
- If ever implemented, require typed confirmation and maybe manual UI confirmation.

## Absolute non-goals

- No disk format.
- No wipe/free-space wipe.
- No recursive deletion of root/system/user folders.
- No deleting hidden/system files by voice.
- No deleting secrets based on AI guess.
- No running `rm -rf`, `del /s`, `format`, or PowerShell deletion generated by LLM.

## Safety model

Destructive file actions are `high_risk_confirmation`.

Rules:

- Dry-run always first.
- Confirmation always required.
- Exact file list or summary required.
- Use Recycle Bin/quarantine first.
- No ambiguous target.
- No broad path.
- No wildcards in first version.
- No voice-only permanent delete.

## Provider abstraction

```csharp
public interface IDestructiveFileActionService
{
    Task<DeletePreview> PreviewRecycleAsync(DeleteRequest request, CancellationToken cancellationToken);
    Task<DeleteResult> MoveToRecycleBinAsync(DeleteExecutionRequest request, CancellationToken cancellationToken);
    Task<RestoreResult> RestoreAsync(string deletionId, CancellationToken cancellationToken);
}
```

public sealed record DeleteRequest(

## Confirmation design

Bad confirmation:

> "Delete it?"

Good confirmation:

```text
I found 3 files to move to the Recycle Bin:
- C:\Users\jarno\Downloads\old-log-1.txt
- C:\Users\jarno\Downloads\old-log-2.txt
- C:\Users\jarno\Downloads\old-log-3.txt
Total size: 14 MB
This is reversible from the Recycle Bin.
Say "move these 3 files to the recycle bin" to confirm.
```

For folders:

```text
This is a folder and may contain many files. I won't delete folders recursively yet.
```

For permanent delete:

```text
I don't support permanent deletion yet. I can move files to the Recycle Bin instead.
```

## Blocked paths

Always block:

- drive roots,
- user profile root,
- Windows directory,
- Program Files,
- AppData except Merlin-owned cache if explicit,
- `.git` folders unless a narrow cleanup recipe exists,
- any path outside allowed folders,
- any path matching sensitive patterns.

## Routing examples

Should route to destructive file action:

- "delete this file"
- "remove those logs"
- "clean up old temp files"
- "move that to the recycle bin"
- "wipe this folder" -> route and refuse dangerous wipe.

Should not route to destructive file action:

- "show me files" -> file access.
- "open this file" -> application/file open capability.
- "remove browser mapping" -> browser mapping tool if it affects config, not filesystem delete.

## Dry-run behavior

Every destructive action begins with a dry-run preview:

1. Resolve target.
2. Validate target scope.
3. Count files/folders.
4. Compute total size.
5. Check hidden/system/sensitive flags.
6. Produce warnings.
7. Stage confirmation.

No preview, no delete.

## Restore behavior

If using quarantine:

- Store deletion id.
- Store original path.
- Store new path.
- Store timestamp.
- Support restore latest deletion.

If using Recycle Bin:

- Restore support may depend on API; if not reliable, say restore must be done manually through Recycle Bin.
